# Self-custody — how DigiByte Wallet handles your keys

DigiByte Wallet is a **self-custodial** wallet. Your seed phrase, your
private keys, and the actual signing of transactions all happen on **your
device**, in **your browser**. The server side of `dgbwallet.app` is a
static asset host — it serves the app, then steps out of the way.

This document is the short, plain-English version. For the line-by-line
audit with file paths, see
[`docs/walletscrutiny-self-eval.md`](docs/walletscrutiny-self-eval.md).

---

## The custody promise, in one paragraph

When you create a wallet, your **12-word seed phrase is generated by code
that runs inside your browser**, encrypted with a password you choose, and
stored in your browser's IndexedDB. The server never sees the seed,
your password, or any private key. When you send a transaction, signing
happens **inside the same browser** using a cryptography library compiled
to WebAssembly. The only thing the server (and the block explorer) ever
see are the same things anyone watching the public DigiByte chain can
already see: addresses and signed transactions.

If we get hit by a bus, **your funds are still yours.** Restore the seed
phrase you backed up into any other DigiByte BIP84 wallet — Sentinel,
Trezor, a fresh copy of this app cloned from the source — and you're back
in business.

## Where keys live

| Stage | What happens | Where it happens |
|-------|--------------|------------------|
| **Generate** | A 12-word BIP39 mnemonic is created using the browser's secure random source. | In the browser, by C# code compiled to WebAssembly. |
| **Encrypt** | You pick a password. The mnemonic is encrypted with **AES-256-GCM**; the password is stretched with **PBKDF2-SHA256, 100 000 iterations**. | In the browser, via the WebCrypto SubtleCrypto API. |
| **Store** | The encrypted blob lands in your browser's **IndexedDB** under the `digibyte-wallet` database. | On your device, in your browser's storage sandbox. |
| **Derive** | When you unlock, the seed is decrypted *into memory only* and used to derive BIP84 keys at `m/84'/20'/0'/change/index`. | In the browser. |
| **Sign** | Transactions are signed with secp256k1 / ECDSA via NBitcoin (the same library Bitcoin Core developers use). | In the browser, inside the WebAssembly sandbox. |

There is **no point in the flow** where the seed, the password, or a raw
private key crosses the network to our servers. There is no cloud backup,
no escrow, no key recovery service. That's the deal — losing your seed
phrase means losing the funds.

## What the server actually sees

`dgbwallet.app`'s backend is plain static asset hosting. From the
server's point of view, you are downloading a `.wasm` file and a handful
of `.js` / `.css` / `.png` assets. After that, all the wallet's
runtime traffic goes to:

- **`digiexplorer.info`** / **`digibyteblockexplorer.com`** — block
  explorers. They see addresses you ask about (which are already public
  on chain) and signed transactions you broadcast (also public). They
  don't see your seed.
- **`api.coingecko.com`** — fiat price feed. Receives no wallet-specific
  data.
- **`api.github.com`** — release/version checks. Receives no
  wallet-specific data.
- **`googletagmanager.com`** — Google Analytics. We use this to count
  pageviews and broad usage. It's not loaded on the signing path; we do
  not send wallet contents, addresses, or any cryptographic material.
  See [WalletScrutiny self-eval §2.2][wsse-22] for the trade-offs we've
  made here.

Compare with your browser's devtools network tab during a normal session
— you'll see exactly these endpoints and nothing else.

## Verify it yourself

You don't have to take our word for it. The audit-friendly version:

1. **Open devtools → Application → IndexedDB → `digibyte-wallet`.** You
   should see your encrypted blob and nothing in plain text. The blob
   format is `salt(16 bytes) ‖ iv(12 bytes) ‖ ciphertext+tag` — see
   [`crypto.js`](src/DigiByte.Web/wwwroot/js/crypto.js).
2. **Open the Network tab and send a test transaction.** Watch what
   leaves your browser. You'll see the *signed* transaction going to a
   block explorer's broadcast endpoint. You will not see your seed,
   your password, or any private key.
3. **Read [`crypto.js`](src/DigiByte.Web/wwwroot/js/crypto.js) and
   [`storage.js`](src/DigiByte.Web/wwwroot/js/storage.js)** — both are
   under 100 lines and call only standard browser APIs.
4. **Trace signing** in
   [`src/DigiByte.Crypto/Transactions/TransactionBuilder.cs`](src/DigiByte.Crypto/Transactions/TransactionBuilder.cs)
   — you'll find no `HttpClient` call carrying key material, only the
   NBitcoin `BuildTransaction(sign: true)` line that does the work
   in-process.

## What if `dgbwallet.app` goes away?

You still control your funds. Two paths:

- **Restore your seed in another wallet.** Any DigiByte wallet that
  supports BIP39 mnemonics + BIP84 derivation (`m/84'/20'/0'`) will pick
  up the same keys and balances.
- **Run a copy of this wallet yourself.** This repo is open source.
  Clone it, build with the pinned .NET SDK in
  [`global.json`](global.json), and serve the `wwwroot/` output from any
  static host. Reproducible-build flags are set in
  [`Directory.Build.props`](Directory.Build.props) so a build from the
  same commit produces the same output.

The wallet is also a **PWA** — once installed, it works offline. The
signing flow doesn't need our backend.

## What we promise, what we don't

We **promise** that:

- We do not collect, transmit, or store your seed phrase, password, or
  private keys. Architecturally we *can't* — the code that touches them
  runs in your browser, never on our infrastructure.
- The server-side of `dgbwallet.app` does not have an endpoint that
  accepts seed phrases or private keys. There is nothing for an
  attacker who compromises the server to take.
- If you find a way to break any of the above, we want to hear about it
  — see [SECURITY.md](SECURITY.md).

We **don't** promise:

- That your device is uncompromised. A keylogger, a malicious browser
  extension, or someone reading over your shoulder can defeat any
  wallet. Hardware wallets exist for exactly this reason and we
  recommend them for amounts where loss would hurt.
- That browser storage is forever. Browsers can clear IndexedDB on
  device cleanups, profile resets, or under storage pressure. **Always
  back up your seed phrase offline.** Paper, metal, two locations.
- That the page you're on is actually `dgbwallet.app`. Phishing sites
  exist. Bookmark the real URL, check your address bar, and never paste
  your seed into anything that didn't come from a source you trust.
